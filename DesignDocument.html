This design document is for a program designed to crack password-protected .zip folders by iterating through a list of passwords pulled from a .txt file, a brute force attack iterating through passwords generated by the itertools module product method, or both.

Outside of the input prompt and the assignment of that input to the method variable, this program operates entirely under the method() function, which consists of a four-branch decision tree that calls other functions depending on the value of the method variable. Almost every function in this program is recursive in order to give the user a chance to try something else if their chosen method fails, or if they’ve provided an invalid input.

method == "1": the open_zip_file() function will be called to pass the variable zip_file for the following brute_force_crack(zip_file) method. The prompt() method is then called to ask the user whether or not they’d like to try to crack another file, or if they’d like to quit the program.

method == "2": open_zip_file() and open_dict_file() are called to pass variables for the dict_crack(zip_file, dict_file) method, respectively. The prompt() method is then called.

method == "3": open_zip_file() and open_dict_file() are called to pass variables to dict_crack(zip_file, dict_file). zip_file is also passed to brute_force_crack(zip_file). The prompt() method is then called.

method == "4": quit().

Unfortunately, users must manually check to see if their .zip folder was successfully cracked. The program instructs them to do this. The other issue is that if the wrong file type is provided for open_zip_file() and open_dict_file(), it will cause an error in the program that I’m not sure how to account for in the code to avoid. Neither of these errors will crash the program, but it will require the user to restart it to attempt another crack.
